# Time Assistant — On-Demand Skill

## What this skill is for

Three routines (`morning-brief`, `evening-review`, `weekly-review`) handle scheduled touchpoints. This Skill handles everything the routines don't:

1. **Ad-hoc classification** — "classify this meeting I'm about to accept"
2. **Planning sessions** — "help me structure next week for 85% HVT"
3. **Deep-dives** — "why has my strategic client time been dropping?"
4. **Rule management** — "add a rule: any event with 'prospecting' in title is HVT/BD"
5. **Corrections** — "the Szymon call I had Tuesday was HVT not LVT, reclassify and remember"

## How it works

**Tenant resolution** (`engine.memory.resolve_store`): the active tenant is determined by, in order: the `TIME_ASSISTANT_TENANT` env var → a single registered tenant in `skill/tenants.json` → the default tenant ID `"rimas"` (which falls back to `MEMORY_REPO_PATH` for backward compatibility). Each tenant maps to its own git-backed memory store.

**Config loading** (`engine.config.load_tenant_config(store_path)`): reads `config.json` from the tenant's store, returning a `TenantConfig`. The framework (categories, targets, weights) and enabled modules all come from this file — nothing is hardcoded.

**Profiles** (`engine.config.load_profile(name)`): named JSON profiles in `profiles/` provide baseline frameworks. The `knowledge-worker` profile is the default. The `consultant` profile extends it with the `client_budget` module. A tenant's `config.json` sets `"profile"` and can override individual keys.

## Framework

**HVT / LVT is the default** (`knowledge-worker` profile). A tenant may define completely custom categories and targets in their `config.json` under `"framework"`. The engine reads whatever is there.

Default HVT categories (target ≥ 80%):
- Strategic client work
- Product building
- Content / personal brand
- Learning / R&D
- Business development

Default LVT categories (hard cap ≤ 20%):
- Email triage & responses
- Internal / status meetings
- Admin, invoicing, bookkeeping
- Reactive chat / Slack
- Support & troubleshooting
- Context switching penalties

## Integrations

Available integrations are listed in `integrations/registry.json` (auto-generated by `integrations/registry.py`). Each adapter implements the `Adapter` ABC from `integrations/base.py` and exposes a uniform `fetch(start, end)` → list of normalized records (`TimeRecord` / `BiometricRecord` / `ActivityRecord` from `engine/records.py`).

Current adapters: `timeular`, `oura`, `strava`, `toggl`.

To fetch data: `integrations.registry.get_adapter(id).fetch(start, end)`.

To see what's available at runtime: `integrations.registry.available()` or inspect `integrations/registry.json`.

To add a new integration, follow `integrations/_contract.md` — no changes to engine code required.

## Credentials

Credentials are resolved via `engine.credentials.get_secret(name)` / `get_secrets(names)`. The provider is set by `TIME_ASSISTANT_CRED_PROVIDER` ∈ `{env, bitwarden, keychain, keystore}` (default: `env`).

**New installs** use `keystore` as `cred_provider` (set automatically by `onboarding.setup.setup(...)`). The keystore provider uses the OS Keychain on macOS and a permission-locked file on Linux/Windows — no external vault required. The tenant registry is stored at `engine.paths.user_config_dir()/tenants.json`.

- **`env`** (default): reads directly from environment variables. Suitable for CI or any environment where secrets are injected at launch.
- **`keystore`** (default for new installs): uses the OS Keychain on macOS (`security find-generic-password`) and a permission-locked JSON file (`secrets.json`, mode 0600) on Linux/Windows. No external vault required.
- **`bitwarden`**: fetches secrets from a Bitwarden-compatible vault via the `bw` CLI. Requires `BW_SESSION` to be set (unlock with `bw unlock --raw` before calling). Vault item names must match the credential env-var names exactly.
- **`keychain`**: reads directly from macOS Keychain via `security find-generic-password` (bypasses the keystore abstraction).

**Rules for all providers:**
- Fetch only the secrets needed for this turn.
- Never echo the resolved secret value to the transcript.
- If a required item is missing, stop and tell the tenant — don't fall back to hardcoded values.

The authoritative credential names for each integration live in `integrations/registry.json` under each manifest's `auth_fields`.

## Memory / tenant store

Each tenant has their own git-backed store (per-tenant repo). Key files:

**Classification + context:**
- `rules.json` — classification rules (seed + learned + client + project patterns)
- `customers.json` — billable client monthly allocations + focus days
- `projects.json` — project/learning weekly allocations
- `day_structure.json` — ideal weekday template (time blocks)
- `template_rules.json` — research-backed drift flags
- `reminders.json` — items to remember, deadlines, pending follow-ups, daily-flag habits
- `corrections_inbox.md` — feedback queue (pending + processed)
- `config.json` — this tenant's profile + framework + enabled modules (see `config.example.json`)

**Scoring state:**
- `daily_scores.json` — historical daily scores
- `weekly_scores.json` — weekly aggregates
- `weekly_archive/YYYY-WWW.md` — past weekly reviews

Before answering any analytical question, read the relevant files. Don't guess.

## Onboarding a new user ("set up my time assistant")

When the user says **"set up my time assistant"** (or no tenant resolves — a first run), run this **resumable** conversational wizard. The wizard is warm, plain-English, one step at a time, with every step skippable.

**Interpreter:** run every helper below with the resolved interpreter — never a bare `python3`:

```bash
PY="${TIME_ASSISTANT_PYTHON:-python3}"
```

Then invoke steps as `"$PY" -c "from onboarding.setup import next_step; ..."`. The Phase 2 Setup-hook bootstrap exports `TIME_ASSISTANT_PYTHON` pointing at a guaranteed interpreter (a system Python ≥ 3.10, or a downloaded standalone Python); until then it falls back to `python3`. If the wizard needs the config-dir path, get it with the resolved interpreter: `"$PY" -c "from engine.paths import user_config_dir; print(user_config_dir())"`.

**Resuming an interrupted setup:** call `onboarding.setup.next_step(store_dir)` to find the first incomplete step. Re-running resumes at that step and never duplicates work. Track completions with `onboarding.setup.mark_step_done(store_dir, "<step-id>")` after each step.

**Batched consent — ask once at the start:** Before running any read/write commands, opening data folders, or reading calendar/integrations/rules, ask the user **once** for broad permission — e.g. "To set you up I'll need to read your data folder and run a few quick read commands (calendar, integrations, rules). OK to proceed and allow access to TimeAssistant/…?" A single "yes, allow access" covers all subsequent read commands in this session. Before each remaining individual step prompt, add one plain-language sentence of *why* that command runs — this prevents a dozen scary-looking permission prompts in a row.

**Steps:**

1. **Identity** (`identity`) — ask the user's name; detect timezone from Bash (`date +%Z` or system locale) and offer it to confirm; ask their working days.

2. **Framework** (`framework`) — pick a time-management profile in plain language: *Knowledge-worker* (default, HVT/LVT) or *Consultant* (adds client budget tracking). No jargon.

3. **Store** (`store`) — the default location is `engine.paths.default_data_dir()`; offer an alternative (e.g. a Dropbox/iCloud/OneDrive folder they name — just ask, don't auto-scan). Then call:
   ```bash
   "$PY" -c "from onboarding.setup import setup; setup('<name>', '<tz>', <working_days_list>, '<profile>', store_dir=<chosen_path_or_None>, modules=<modules_from_profile>)"
   ```
   Pass `store_dir=None` to use the default per-OS data dir; pass the chosen folder path when the user picks one. `working_days` is a list (e.g. `['Mon','Tue','Wed','Thu','Fri']`) and `modules` is the module list from the selected profile. After success, reveal the folder: `open <dir>` on macOS, `xdg-open` on Linux, `explorer` on Windows.

4. **Calendar — recommended first source** (`calendar`) — connect **Google Calendar** through Claude's own connector UI (no token paste required). This single connection makes the assistant immediately useful and is fully self-contained.

5. **Optional enrichments** (`enrichments`) — for each provider in `onboarding.connect.PROVIDER_FIELDS` the user wants (Oura, Timeular/EARLY, Toggl): open **exactly one** provider page at a time, using the specific sub-URL for token/profile creation (e.g. Toggl → Profile page, not a generic Integrations hub; never open multiple provider tabs at once). Wait for the user to complete that step, then move to the next. Open with `open`/`xdg-open`/`explorer`, then tell them to run the hidden-input helper **themselves** with the `!` prefix (keeps the secret out of the chat, command previews, and terminal history).

   **⚠ Paid-plan warning — say this before opening the page:** Timeular integrations and the EARLY API each require a **paid plan of that service**. Before opening Timeular or EARLY setup pages, say: "Note: this integration requires a paid [Timeular / EARLY] subscription. Skip if you don't have one — you can add it any time." Let the user skip without friction.

   **Oura "no data yet" = normal sync delay, not an error:** If Oura returns no readiness data immediately after connecting, tell the user: "Your ring is still syncing — this is normal and usually resolves within a few minutes. The assistant will pick up data on the next run." Do not frame this as a connection error.

   First **determine the plugin's install directory** and give the user the command with the **FULL absolute path** to the helper — do NOT rely on `${CLAUDE_PLUGIN_ROOT}` being set in the user's `!` shell (it usually is not, and the path would expand to `/skills/...` → "no such file"). For example, after resolving the absolute path:

   ```
   ! "$TIME_ASSISTANT_PYTHON" "/abs/path/to/skills/time-assistant/onboarding/store_token.py" oura
   ```

   The helper hides input, validates against the provider's API, and stores in the OS keystore. On failure it reports without storing and the user can re-run. All enrichments are skippable — tell the user they can add them anytime.

   **Do NOT ask the user to paste tokens into the chat and never put a token in a command you run.** Secrets are entered only through these `!`-run hidden prompts.

   - **Strava (advanced)** — Strava needs an API app and an OAuth round-trip, so it is a few more steps; offer it only if the user wants training-load data. **⚠ Paid-plan warning:** the Strava API requires a paid Strava subscription for extended data access — warn the user before proceeding and let them skip. Tell them to run the Strava connector **themselves** with the `!` prefix (hidden client id/secret + on-device OAuth). As above, substitute the **FULL absolute path** to the script — do NOT rely on `${CLAUDE_PLUGIN_ROOT}` in the user's `!` shell:

   ```
   ! "$TIME_ASSISTANT_PYTHON" "/abs/path/to/skills/time-assistant/onboarding/strava_connect.py"
   ```

   The script reads Client ID and Client Secret via hidden prompts, opens the Strava authorization page, catches the OAuth callback on a localhost listener, exchanges for a refresh token, and stores all credentials in the OS keystore — no values are shown or logged. All on-device, no backend. If they do not use Strava, skip it.

   **Security note:** the assistant must never request a secret in chat or place one in a command — secrets are entered only through these `!`-run hidden prompts.

6. **First brief** (`first_brief`) — generate the first morning brief immediately so the user sees the assistant working before they leave the wizard.

   **Optionally, offer to schedule a daily brief.** Ask if they would like the brief to run automatically each morning and at what time. If yes: `art = onboarding.schedule.artifact_for(<platform>, hour, minute)`; show them exactly what will be added; on macOS/Linux call `onboarding.schedule.install(<platform>, art)` after they agree; on Windows show `art["content"]` for them to paste. Always show what is being installed first — never modify their system silently. When asking the user to confirm any action (e.g. "Install this?" or "Shall I schedule this?"), explicitly restate exactly what to type — for example: "Type `yes` to install, or press Enter to skip." Claude Code's session-rating survey can appear alongside the prompt and obscure the original question, so restating makes the confirmation unambiguous. Skippable.

After each step, call `onboarding.setup.mark_step_done(store_dir, "<step-id>")`. Note: progress is persisted only from the **store** step onward — before the store exists there is nothing to write, and `mark_step_done` safely no-ops.

**Credentials / tenant-store notes for new installs:**

- Secrets are stored via the `keystore` provider (macOS Keychain on macOS; a permission-locked file on Linux/Windows). This is recorded as `"cred_provider": "keystore"` in `config.json`.
- The tenant registry for new installs lives at `engine.paths.user_config_dir()/tenants.json`. The `onboarding.setup.setup(...)` call creates this entry automatically.

## Trigger patterns & behaviors

### "Analyze my week" / "How am I doing this week?"

1. Read last 7 days from `daily_scores.json`
2. Pull today's time-tracking entries for current-day context (via configured time-tracking adapter)
3. Compute week-to-date HVT ratio
4. If Friday or later, offer to run the full weekly review
5. Otherwise give a short mid-week snapshot

### "Plan my week" / "Plan next week"

This is a Socratic planning session, not a scheduler:

1. Ask what the 2–3 highest-leverage outcomes are for the week
2. Ask what HVT blocks are non-negotiable (client commitments, content cadence)
3. Ask what LVT batches the tenant wants (when email lives, when invoicing happens)
4. Then — and only then — propose a block structure with explicit HVT% target
5. Ask if the tenant wants it pushed to Google Calendar (use Calendar MCP to create events)

Never autogenerate a week. Force the tenant to name priorities first.

### "Classify this event: [description]"

1. Read `rules.json`
2. Check for pattern match
3. If no match, classify using the tenant's framework; explain reasoning in 1–2 sentences
4. Ask: "Save as a permanent rule?" before writing to `rules.json`

### "Why am I drifting on [category]?"

1. Read last 4 weeks from `daily_scores.json`
2. Identify trend in the named category
3. Look for correlations: specific days of week, specific clients, surrounding events
4. Present findings; don't prescribe unless asked

### "Reclassify [event]" / "[event] was actually HVT"

1. Update the specific day's row in `daily_scores.json` (or note correction if the event predates scoring)
2. Offer to create a permanent rule so this doesn't repeat
3. Commit with message: `correction: {event} → {new label}`

### "Show me my rules" / "Edit rule"

Read `rules.json` and display. For edits, confirm before writing. Rules have this shape:

```json
{
  "id": "rule_042",
  "pattern": "Acme architecture",
  "client": "Acme",
  "label": "HVT",
  "category": "strategic",
  "confidence": 1.0,
  "created_at": "2026-04-18",
  "source": "user"
}
```

### "What did the routine say this morning?" / "What was today's score?"

Pull from `daily_scores.json` and summarize. Don't re-run the routine.

### "How's today looking?" / "Run morning brief" (on-demand local mode)

Produces a full brief identical to the routine, but run locally on request:

1. Pull **today's Google Calendar** events (tenant's timezone from `config.json`)
2. Pull **last 7 days time-tracking entries** via the configured time-tracking adapter. **TIMEZONE RULE (mandatory):** time-source timestamps (Toggl, etc.) are UTC — convert every timestamp to the tenant's timezone BEFORE bucketing by day or computing hours: `engine.timeutil.to_zone(ts, config.user.timezone)`. Failure to convert causes entries to appear in the wrong day (e.g. 04:10 UTC shown instead of 06:10 CEST). Then compute HVT%, unclassified %.
3. Pull **Oura readiness** (today + 7-day trend) if `oura` is in tenant's `config.integrations`
4. Pull **last 7 days Strava load** (suffer, hours, HIIT count) if `strava` is in tenant's `config.integrations`
5. Apply `rules.json` to classify events; apply `template_rules.json` for drift flags
6. Cross-ref `customers.json` focus day + `projects.json` allocations for the day
7. **Client budget tracker** — only when the `client_budget` module is enabled (e.g. the `consultant` profile). Enabled check: `modules.client_budget.is_enabled(config)`. When enabled, render a table of every allocated client from `customers.json` with month-to-date tracked vs this month's budget:
   - **Budget** = current-month allocation, resolved with fallback via `modules.client_budget.resolve_budget(customer, month_key)`: `<month_key>_allocated_hours` if present, else the most recent confirmed `*_allocated_hours`, else `allocated_hours`.
   - **MTD tracked** = sum of all time-tracking block hours tagged with that client this month. Pre-aggregate into a `{tag: hours}` dict before calling the table function.
   - Table via `modules.client_budget.budget_table(customers, mtd_by_tag, month_key, pct_month_elapsed)` where `mtd_by_tag` is a `dict[str, float]` of hours-by-client-tag and `pct_month_elapsed` is the fraction of the current month elapsed (e.g. `day/days_in_month`). Returns `list[dict]`; each row has keys: `tag, focus_day, budget_h, mtd_h, remaining_h, pct_used, pace`. Columns: Client · Focus day · Budget (h) · MTD (h) · Remaining (h) · % used · pace flag.
   - Pace flag: compare % of month elapsed vs % budget used; flag clients trending to under- or over-deliver. Call out today's focus-day client explicitly if it's at 0h.
   - This section is **mandatory in every morning brief when the module is enabled**. Keep it to one compact table.
8. Produce: snapshot, today's classified table, client budget tracker (if enabled, step 7), top 3 HVT moves, template drift flags, biometric-driven recommendation

### "Evening review" / "Run evening review" (on-demand local mode)

Produces the end-of-day review. Steps:

1. Pull **today's tracked entries** via the configured time-tracking adapter (use `--json` and decode manually; the text formatter can crash on null notes). **TIMEZONE RULE (mandatory):** timestamps are UTC — convert every entry's timestamp with `engine.timeutil.to_zone(ts, config.user.timezone)` BEFORE bucketing by day or displaying times. Skipping this causes entries to appear shifted (e.g. 04:10 UTC instead of 06:10 CEST).
2. Classify each block (HVT/LVT or tenant's framework) via `rules.json`; surface any genuinely ambiguous block (one that swings the score materially) with `AskUserQuestion` rather than guessing.
3. Compute the day: tracked hours, HVT%, score (`hvt_pts`/`adherence_pts`/`frag_pts`), boundary-breach flag. Scoring logic lives in `engine/score.py` (`score_day`).
4. **Client budget tracker** — only when the `client_budget` module is enabled (same check and rendering as morning brief). When enabled, render the table from `customers.json` MTD tracked vs this month's budget using today's scored blocks:
   - **Budget**: resolved with the same fallback as morning brief.
   - **MTD tracked**: sum of all `actual_blocks` minutes tagged with that client across this month's `daily_scores.json` entries (include today's blocks just scored).
   - Columns: Client · Focus day · Budget (h) · MTD (h) · Remaining (h) · % used · on/off-pace flag.
   - Pace flag: flag clients trending to under- or over-deliver. Call out today's focus-day client if it got 0h.
   - This section is **mandatory in every evening review when the module is enabled**.
5. Surface reminders due/overdue + any `recurring` habit that fired today; offer to mark actioned items `completed`.
6. Write today's entry to `daily_scores.json`, commit + push.

### "Remind me" / "What's on my mind?" / Auto-surface reminders

Read `reminders.json`. Surface items where:
- `status="pending"` AND `due` is within next 7 days (lookahead window)
- `status="pending"` AND `due` is overdue (any time in past 30 days)
- `recurring=*` items get checked daily, flagged when their condition fires (e.g. sport untagged, boundary breached)

Morning brief automatically surfaces these in a "Reminders" section. Evening review checks if anything was actioned today and offers to mark `completed` (move to `completed[]` array, don't delete).

To add a reminder mid-conversation: write the JSON entry directly + commit.

To mark done: tenant confirms → flip status to completed.

### "Check email load" / "What's blocking my inbox today?"

Use Gmail MCP (`search_threads`, `get_thread`):
1. Count `is:unread` in last 24h
2. Identify client-tagged threads (match against `customers.json` + `rules.json` patterns)
3. Flag if inbox growth suggests moving LVT email batch earlier

## Tools & dependencies

- **Tenant memory store** — read/write via `engine.memory` (`read_json`/`write_json`) + local git clone; push via `gh` CLI or git
- **Google Calendar MCP** — event read, event create (for "push plan to calendar")
- **Gmail MCP** (`search_threads`, `get_thread`, `create_draft`, labels) — email load signal + draft batching
- **Adapters** — `integrations.registry.get_adapter(id)` returns the adapter for a given integration ID; call `.fetch(start, end)` for normalized records. See `integrations/registry.json` for available IDs.

## Behaviors to avoid

- Don't produce a full routine-style brief on casual queries. "How am I doing?" deserves 2–3 sentences, not a 200-line dashboard.
- Don't moralize. The tenant is optimizing a system.
- Don't invent rules. Only promote a classification to a permanent rule with explicit tenant confirmation.
- Don't assume the tenant wants a plan pushed to Calendar unless they said so.
- If the tenant asks about the routines themselves (scheduling, how to edit), refer them to `onboarding/SETUP.md` — this Skill is about time analysis, not routine admin.
- Don't show the client budget tracker if `client_budget` is not in the tenant's enabled modules.
